# Hybrid BM25 + Vector Retrieval with RRF and Cross-Encoder Reranking

- Timestamp: 2026-04-10 20:46:34 SGT
- Commits: `d066794`

## Summary

Introduced a hybrid retrieval layer on top of the existing Flask + Solr stack.
BM25 (eDisMax) and dense-vector (kNN) retrieval run in parallel against the same
Solr index, are fused via Reciprocal Rank Fusion, then reranked by a cross-encoder
service. The existing `GET /` endpoint, filters, facets, and NLP pipeline are
preserved unchanged. All new semantic stages degrade gracefully to BM25-only when
the embedding or reranker service is unreachable.

## Changes

### `hybrid_search.py` — new file

Core retrieval module. Three public classes plus a standalone RRF function:

**`EmbeddingClient`**
- HTTP adapter for the embedding service at `:8081` (configurable via `EMBEDDING_URL`)
- `embed(texts)` — `POST /embed` with `{"texts": [...]}`, accepts `embeddings`, `vectors`, or `data` keys in the response
- `embed_query(text)` — single-text convenience wrapper
- `embed_batch(texts, batch_size)` — chunks texts into batches of `EMBED_BATCH_SIZE` (default 32); individual `None` entries on partial failures
- Returns `None` on any network or parsing error; never raises

**`RerankerClient`**
- HTTP adapter for the reranker service at `:8082` (configurable via `RERANKER_URL`)
- `rerank(query, candidates, top_k)` — `POST /rerank` with `{"query": "...", "documents": [...]}`
- Handles both flat `[float]` and `[{"index": N, "score": F}]` score response shapes
- Returns `None` on failure; never raises

**`reciprocal_rank_fusion(lexical_ids, vector_ids, rrf_k)`**
- Standard RRF: `score += 1 / (rrf_k + rank)` for each list
- Deduplicates by document id (union of both candidate sets)
- Returns `[(id, score)]` sorted descending

**`HybridSearchService.search()`**
Five-stage pipeline returning `(results, facets, num_found, RetrievalInfo)`:

| Stage | Description |
|---|---|
| 1. Lexical | eDisMax with all existing `qf`/`pf`/`bq`/`boost` params; fetches up to `HYBRID_LEXICAL_K=100` candidates and full facet counts |
| 2. Vector | Embeds query via `EmbeddingClient`; issues `{!knn f=content_vector topK=N}` Solr query with the same `fq` filters; fetches up to `HYBRID_VECTOR_K=100` candidates |
| 3. RRF | Fuses the two candidate id lists; deduplicates |
| 4. Rerank | Passes the top `HYBRID_RERANK_K=50` fused candidates to `RerankerClient` with `search_text` as the document text |
| 5. Highlight fetch | Issues a final Solr query filtered to the reranked id set to retrieve highlighting snippets using the lexical/NLP query |

Degradation rules:
- Vector stage fails → `mode="lexical"`, `degraded=True`, warning surfaced in UI
- Reranker fails → serve RRF-ordered results, `degraded=True`, warning surfaced in UI
- Both fail → pure BM25 behaviour, no warning

**`RetrievalInfo`** dataclass / `as_dict()` exposes:
`mode`, `degraded`, `warnings`, `lexical_hits`, `vector_hits`, `fused_hits`,
`reranked_hits`, `latency_ms` (per-stage breakdown).

**Config env vars** (all have defaults):

| Variable | Default | Purpose |
|---|---|---|
| `EMBEDDING_URL` | `http://localhost:8081` | Embedding service base URL |
| `RERANKER_URL` | `http://localhost:8082` | Reranker service base URL |
| `HYBRID_LEXICAL_K` | `100` | Max lexical candidates |
| `HYBRID_VECTOR_K` | `100` | Max vector candidates |
| `HYBRID_RRF_K` | `60` | RRF constant k |
| `HYBRID_RERANK_K` | `50` | Candidates passed to reranker |
| `SEARCH_ROWS` | `20` | Final results returned to UI |
| `EMBED_BATCH_SIZE` | `32` | Texts per embedding HTTP call |
| `EMBEDDING_DIM` | `384` | Vector dimension (also used in schema) |

### `app.py` — updated

- Imports `EmbeddingClient`, `RerankerClient`, `HybridSearchService`, `RetrievalInfo`, `SEARCH_ROWS` from `hybrid_search`
- Instantiates the three service singletons at module load time
- Replaces the direct `requests.get(SOLR_URL, ...)` block with `_hybrid.search(...)` — all NLP preprocessing, filter building, and `qf`/`pf`/`bq` construction are unchanged upstream of the call
- Passes `query_text=q` (original pre-NLP query) to the hybrid service so the embedding uses the user's natural-language input, not the Solr-processed form
- Adds `retrieval_info` to the `render_template` call
- Surfaces degradation warnings through the existing `error` banner

### `schema_add_fields.json` — updated

Added two new schema directives:

**`add-field-type`** — defines the `knn_vector` field type:
```json
{
  "name": "knn_vector",
  "class": "solr.DenseVectorField",
  "vectorDimension": "${EMBEDDING_DIM:384}",
  "similarityFunction": "cosine",
  "knnAlgorithm": "hnsw",
  "hnswMaxConnections": 16,
  "hnswBeamWidth": 100
}
```

**`add-field`** — adds `content_vector` using the new type:
```json
{"name": "content_vector", "type": "knn_vector", "stored": true, "indexed": true}
```

The 19 existing fields and all copy-field directives are unchanged.
The `vectorDimension` defaults to 384 but is overridable via the `EMBEDDING_DIM` env var at schema-apply time.

### `scripts/prepare_solr_docs.py` — updated

- Added `content_vector: Optional[list[float]] = None` field to `SolrDoc`
- `SolrDoc.to_dict()` omits `content_vector` when `None` so documents without embeddings are still accepted by Solr
- Added `embed_docs(docs)` function:
  - Filters out docs with empty `search_text` before embedding
  - Calls `EmbeddingClient.embed_batch()` in configurable batches
  - Writes embedding into `docs[i]["content_vector"]` in-place
  - Logs success/failure counts; partial batch failures leave the affected docs without a vector
  - Prints a clear `[INFO]` message if the embedding service is unreachable (does not abort the pipeline)
- `prepare_docs()` calls `embed_docs(all_docs)` after deduplication, before writing JSONL

## Notes

- The `content_vector` field must be added to the Solr schema (via `schema_add_fields.json`) and documents must be reindexed with `prepare_solr_docs.py` before vector retrieval activates. Until then, the embedding service returns `None` and the app degrades cleanly to BM25.
- The exact HTTP wire format of `:8081` and `:8082` was not verified against live services; the adapter layer in `EmbeddingClient` and `RerankerClient` tolerates the most common response shapes. Adjust the key names in those adapters if the actual services differ.
- Solr's `{!knn}` query parser requires Solr 9.0+; the repo runs Solr 10.0.0, so this is satisfied.
- Facet counts in the UI always reflect the lexical candidate pool (wider set), not the reranked slice, to give users accurate drill-down numbers.
