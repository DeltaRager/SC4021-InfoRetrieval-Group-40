# Chunk-Based Vector Retrieval

- Timestamp: 2026-04-10 21:54:35 SGT

## Summary

Replaced the single `content_vector`-per-document approach with chunk-based vector indexing inside the existing `reddit_ai` Solr core. Long documents are now split into overlapping chunks before embedding, eliminating embedding failures caused by oversized inputs. Vector retrieval operates on chunk records and collapses back to source documents before RRF fusion and reranking. BM25 fields and the document-oriented UI are unchanged.

## Changes

### `schema_add_fields.json`

Added chunk fields to the Solr schema:

- `doc_id` (string, indexed) ŌĆö stable source-document id; groups all chunks from one post/comment
- `chunk_id` (string, indexed) ŌĆö unique id per chunk record
- `chunk_index` (pint) ŌĆö ordinal position of the chunk within the source document
- `chunk_text` (text_en, stored-only) ŌĆö the text actually embedded for kNN retrieval
- `chunk_count` (pint) ŌĆö total number of chunks produced from the source document
- `chunk_char_start`, `chunk_char_end` (pint) ŌĆö character offsets into `search_text`
- `chunk_vector` (knn_vector, 1024-dim cosine HNSW) ŌĆö dense vector field; replaces the old `content_vector`

The `knn_vector` field type definition is unchanged. `content_vector` is no longer added to new documents (the field type remains in schema for any existing indexed data).

### `scripts/prepare_solr_docs.py`

- Removed `content_vector` from `SolrDoc` dataclass; updated docstring to describe chunk-record output format.
- Added constants `CHUNK_TARGET_CHARS = 1200`, `CHUNK_OVERLAP_CHARS = 200`, `EMBED_MAX_CHARS = 4000`.
- Added `_split_into_chunks(text, target, overlap)` ŌĆö paragraph-aware splitter with sliding-window fallback. Returns `(chunk_text, char_start, char_end)` tuples. Short texts produce exactly one chunk.
- Added `_make_chunk_records(doc)` ŌĆö expands one source-document dict into one or more chunk dicts. The Solr `id` becomes the chunk id (`<doc_id>__c<n>`); the original id is stored in `doc_id`. All source-document metadata is duplicated onto every chunk record so filters and facets work without joins.
- Rewrote `embed_docs(docs)` ŌĆö now embeds `chunk_text` (not `search_text`) and writes `chunk_vector`. Texts exceeding `EMBED_MAX_CHARS` are truncated before the embedding call. Reports chunks embedded, chunks skipped after retry, and source docs with zero vector chunks.
- Rewrote `prepare_docs()` ŌĆö now expands source documents to chunk records before embedding, then writes one JSONL line per chunk. Returns `(source_docs_processed, total_chunks, zero_vector_docs)`.
- Updated `main()` to print the new summary counts.

### `hybrid_search.py`

- Raised `HYBRID_VECTOR_K` default from 100 ŌåÆ 200 to compensate for multi-chunk competition.
- Added `CHUNK_VECTOR_FIELDS` constant ŌĆö `DISPLAY_FIELDS` extended with `chunk_text,chunk_index` for the kNN query.
- Added `FACET_FIELDS`, `_collapse_filter()`, `_json_facet_payload()`, `_parse_doc_facets()` ŌĆö use Solr collapse (`{!collapse field=doc_id}`) and JSON faceting so lexical retrieval already returns one record per source document and facet counts reflect documents rather than chunks.
- Rewrote `_vector_retrieval()` ŌĆö queries `chunk_vector` via kNN; collapses chunk hits to unique `doc_id`s in descending score order (first occurrence = best chunk); returns `(collapsed_docs, best_chunk_texts)` where `best_chunk_texts` maps `doc_id ŌåÆ chunk_text` of the highest-scoring chunk.
- Updated `search()` RRF stage ŌĆö deduplicates lexical hits by `doc_id` before RRF; both lexical and vector sides now contribute doc-level ids to `reciprocal_rank_fusion`. The best chunk text is used as the reranker `search_text` evidence when available.
- Reworked `_lexical_retrieval()` ŌĆö lexical BM25 now runs against the chunked core with a Solr collapse filter on `doc_id`, preventing long documents from consuming multiple lexical candidate slots. The query also switches from classic facet counts to JSON facets with `unique(doc_id)` so `num_found` and sidebar buckets stay document-level.
- Rewrote `_fetch_with_highlighting()` ŌĆö filters by `doc_id:(...)` instead of `id:(...)`, applies a collapse filter so Solr returns one representative record per source document, and removes the old `len(doc_ids) * 5` row multiplier that could silently drop ranked documents when one doc had many matching chunks.
- Rewrote `_finalize_lexical()` ŌĆö deduplicates by `doc_id` before fetching highlights and in the raw fallback path.

These retrieval changes fix three regressions introduced by the initial chunk refactor:

- long documents dominating BM25 lexical candidate slots
- final ranked documents disappearing during the highlight fetch step
- inflated `num_found` / facet counts caused by chunk-level counting

### `tests/test_hybrid_search.py`

Added imports for new ETL helpers. Updated `_make_solr_docs` to produce chunk records with `doc_id`, `chunk_index`, and `chunk_text`.

New test classes:

- `TestSplitIntoChunks` (6 tests) ŌĆö short/empty/long texts, ordered boundaries, overlap prefix, whitespace-only input.
- `TestMakeChunkRecords` (7 tests) ŌĆö single chunk for short docs, multiple chunks for long docs, unique chunk ids, stable `doc_id`, source metadata preserved on every chunk, chunk count consistency.
- `TestChunkCollapse` (8 tests) ŌĆö multiple chunks from same doc collapse to one; best chunk text wins; embed failure returns `(None, {})`; RRF fuses by `doc_id` (duplicate lexical chunk ids collapsed); zero vector chunks degrades cleanly to lexical-only; `fq` filters forwarded to chunk kNN query; reranker receives one `Candidate` per source document.

Updated `TestEmbedDocs` to use `chunk_text`/`chunk_vector` instead of `search_text`/`content_vector`; added test asserting one failing chunk does not drop the whole source document.

Added regression coverage for the post-review fixes:

- lexical retrieval reports document-level `num_found` from JSON facets rather than raw chunk count
- lexical Solr queries include `collapse field=doc_id`
- final highlight fetch uses one collapsed record per requested `doc_id` rather than a heuristic row multiplier

## Notes

- A Solr reindex is required after applying the updated schema: run `schema_add_fields.json` against the `reddit_ai` core, then re-run `prepare_solr_docs.py` to regenerate the JSONL and re-POST to Solr.
- The `content_vector` field type is still declared in the schema for any data already indexed; it is simply no longer populated by the ETL. It can be removed via a schema API call once the core is fully reindexed with chunk records.
- The targeted hybrid-search test suite now passes at 51 tests.
