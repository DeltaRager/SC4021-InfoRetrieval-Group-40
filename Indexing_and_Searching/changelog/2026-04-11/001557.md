# Chunk-Level Reranking with Max-Pool Deduplication

- Timestamp: 2026-04-11 00:15:57 SGT

## Summary

Replaced document-level reranking with chunk-level reranking. Previously, the pipeline collapsed each source document to a single representative chunk before sending candidates to the cross-encoder reranker. Now, all chunks for each fused candidate document are sent to the reranker individually, and scores are max-pooled back to the document level (best chunk score wins). This ensures the reranker evaluates every passage of a document against the query rather than a single arbitrarily chosen chunk.

Also fixed a related issue where `search_text` (the full merged concatenation of all chunks) was being passed to the reranker as the document text for lexical-only candidates. The per-chunk `chunk_text` field is now fetched during lexical retrieval and used preferentially.

## Changes

### `hybrid_search.py`

**Config**

- Added `HYBRID_RERANK_CHUNK_K = 150` (env: `HYBRID_RERANK_CHUNK_K`) — caps the total number of chunks sent to the reranker after expansion. Tune downward if the reranker's context window is exceeded; tune upward to allow more candidates.

**`Candidate` dataclass**

- Added `doc_id: str` field — the stable source-document identity (e.g. `abc123`). `id` is now the chunk-level Solr id (e.g. `abc123__c1`). This separation is necessary because the reranker operates on chunks while all downstream steps (highlighting, result rendering) operate on documents.

**`DISPLAY_FIELDS` / `CHUNK_VECTOR_FIELDS`**

- Added `chunk_text` to `DISPLAY_FIELDS` so lexical retrieval now fetches per-chunk text alongside `search_text`.
- `CHUNK_VECTOR_FIELDS` updated to avoid duplicating `chunk_text` (it is already in `DISPLAY_FIELDS`).

**`HybridSearchService.search()` — Stage 4 rewrite**

Previously:
```
fused doc_ids → one Candidate per doc (best chunk text heuristic) → reranker → sorted docs
```

Now:
```
fused doc_ids → _fetch_chunks_for_reranking → one Candidate per chunk → reranker
             → max-pool by doc_id (best chunk score) → sorted docs
```

- Stage 4a (`chunk_expand`): calls `_fetch_chunks_for_reranking()` to expand the top `HYBRID_RERANK_K` fused doc_ids to all their chunks.
- Stage 4b (`rerank`): sends all chunks to the reranker; after scoring, groups by `doc_id` and keeps the candidate with the highest `rerank_score` per document; sorts docs by that score.
- Degraded fallback (reranker unavailable): reconstructs a doc-level `Candidate` list in RRF order using the in-memory representative records, exactly as before.
- `final_ids` now reads `c.doc_id` instead of `c.id` to pass document identifiers to `_fetch_with_highlighting`.

**`HybridSearchService._fetch_chunks_for_reranking()` — new method**

Fetches all Solr chunk records for a given list of `doc_id`s in a single query (filtered by `doc_id:(...)`). Returns a flat `list[Candidate]`, one entry per chunk, with:
- `id` = the chunk's Solr `id`
- `doc_id` = the source document identity
- `search_text` = `chunk_text` (per-chunk text, bounded by the chunker's target size)
- `display_fields` = the in-memory representative record for the doc (carries title, body, metadata)
- `rrf_score` / `source` inherited from the RRF fusion result

Falls back to one candidate per doc (using the in-memory representative record) if the Solr fetch fails.

**`RerankerClient.rerank()`**

- Removed the `ranked[:top_k]` slice at the end. The caller is now responsible for any top-k slicing (max-pool deduplication happens before the final slice to `SEARCH_ROWS`). All scored chunks are returned sorted by score.

## Notes

- The chunk distribution in the indexed corpus is: median 1 chunk/doc, mean 1.7, max 28. With `HYBRID_RERANK_K=50` fused docs the typical chunk expansion is ~85 chunks — well within `HYBRID_RERANK_CHUNK_K=150` and the reranker's context budget at `RERANKER_MAX_DOC_CHARS=512` chars/chunk.
- No schema or reindex changes required. This is a retrieval pipeline change only.
- No test changes in this entry; the existing test suite continues to pass.
